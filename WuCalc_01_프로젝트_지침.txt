# WuCalc 프로젝트 지침 (이 텍스트를 프로젝트 "지침" 란에 복사-붙여넣기)

## 프로젝트 정의
WuCalc는 명조(Wuthering Waves) 올인원 빌드 계산/시뮬레이션 웹 사이트다. 에코 비교, 무기 비교, 파티 DPS 시뮬레이션, 보스 클리어 가능성 판단, 가챠 확률 계산 등을 한 곳에서 제공하는 것이 목표다.

## 프로덕션 기준 (최우선 원칙)
이 프로젝트는 학습용/데모가 아니다. GCP 서버에서 실제 유저에게 서비스하는 프로덕션 애플리케이션이다. 모든 코드는 아래 기준을 만족해야 한다:
- 실제 서비스 가능한 완성도. 기능 stub, placeholder, 하드코딩된 더미 데이터, "나중에 구현" 주석은 허용하지 않는다. 구현을 요청하면 해당 기능이 실제로 동작하는 코드를 작성한다.
- 프로덕션 레벨 성능 최적화. GCP 인스턴스의 메모리와 CPU를 최소한으로 사용한다.
  - React 컴포넌트: React.memo, useMemo, useCallback을 적절히 사용하여 불필요한 리렌더링 방지
  - 번들 사이즈: dynamic import, tree shaking 활용. 사용하지 않는 라이브러리 import 금지
  - 이미지: next/image로 최적화, 적절한 사이즈와 포맷 사용
  - JSON 데이터: 페이지에서 필요한 데이터만 로드. 전체 캐릭터 데이터를 한 번에 불러오지 않는다
  - SSG(Static Site Generation) 최대한 활용하여 서버 런타임 부하 최소화
- 보안: XSS 방어, 사용자 입력 검증, Content Security Policy 헤더 설정
- 에러 처리: 모든 사용자 인터랙션에 에러 바운더리 및 fallback UI 적용. 계산 엔진에서 NaN/Infinity 발생 시 안전하게 처리
- 접근성: 시맨틱 HTML, aria 속성, 키보드 네비게이션 지원
- SEO: 페이지별 메타태그, Open Graph, structured data. 검색 유입이 수익과 직결되므로 철저하게

## 외부 데이터/API 활용 규칙
- 게임 데이터(캐릭터 스탯, 스킬 배율, 무기, 에코 등)를 직접 타이핑하지 않는다. 공개된 API나 데이터 소스가 있으면 반드시 그것을 활용한다.
- 데이터가 필요한데 프로젝트 지식 파일에 없거나 불확실하면, 웹 검색으로 공개 API/데이터 소스를 찾아서 실제 사용 가능한 것을 적용한다.
- 현재 알려진 활용 가능 소스:
  - Wuthering Waves Fandom Wiki API: 캐릭터, 무기, 에코 데이터
  - hakush.in API: 게임 데이터 (비공식, 커뮤니티 검증)
  - GitHub 오픈소스 프로젝트: wutheringwaves-data 등 커뮤니티 데이터 리포지토리
  - WuWa Tracker 공개 데이터: 가챠 소프트 천장 확률 모델
- 위 소스가 부족하거나 변경되었을 경우, 웹 검색으로 대체 소스를 직접 찾는다.
- API를 사용할 때는 빌드 타임에 데이터를 fetch하여 정적 JSON으로 저장하는 방식을 기본으로 한다(런타임 외부 API 의존 최소화). 단, 실시간성이 필요한 데이터(예: 현재 배너 정보)는 예외.
- 데이터 파싱 스크립트는 scripts/ 디렉토리에 두고, npm script로 실행 가능하게 한다 (예: `npm run sync-data`).
- 외부 소스에서 가져온 데이터는 반드시 TypeScript 타입으로 검증한 후 사용한다. 스키마가 맞지 않으면 빌드 단계에서 에러를 발생시킨다.

## 자율적 문제 해결
- 구현 중 모르는 부분, 라이브러리 사용법, 최신 API 스펙 등이 있으면 스스로 웹 검색하여 해결한다. 추측으로 코드를 작성하지 않는다.
- 게임 메커니즘에 대해 확실하지 않은 부분은 검색으로 최신 커뮤니티 검증 데이터를 확인한 후 구현한다.
- 패키지 버전 호환성 문제가 발생하면 검색으로 해결 방법을 찾아 적용한다. deprecated된 API를 사용하지 않는다.

## 게임 데이터 정확성 (최우선 규칙)
이 사이트는 정보 제공이 핵심 가치다. 잘못된 이름이나 수치는 사이트의 신뢰도를 즉시 파괴한다. 아래 규칙을 반드시 지킨다:
- 캐릭터 이름, 무기 이름, 에코 이름, 세트 이름, 스킬 이름 등 게임 내 고유명사는 반드시 공식 명칭을 사용한다. 추측하거나 기억에 의존하지 않는다.
- 고유명사를 코드/데이터에 쓸 때 확실하지 않으면 반드시 웹 검색으로 공식 명칭을 확인한다. "대충 이런 이름이었던 것 같다"로 작성하지 않는다.
- 한국어/영어 명칭이 다를 수 있다. 양쪽 모두 공식 인게임 표기를 따른다. 번역을 임의로 하지 않는다.
  - 예: 지연(X), 지얀(X) → 기염(O, 한국어 공식명), Jiyan(영어 공식명)
  - 예: 에코 세트도 "Sierra Gale"이 영어 공식명이면 한국어 공식명을 별도 검색한다
- 스킬 배율, 스탯 수치, 에코 메인/서브옵 범위 등 숫자 데이터도 동일 기준 적용. 기억으로 적지 않고, 검증된 소스(Wiki, 데이터마이닝 리포, 인게임 확인)에서 가져온다.
- 데이터 JSON 파일에 오타/오기가 섞이면 계산 결과 전체가 틀어진다. 소스에서 가져온 데이터를 수동으로 수정할 때는 변경 사유를 주석으로 남긴다.

## 개발 환경 규칙
- 실제 서비스는 GCP 리눅스 서버에서 Docker로 운영한다.
- 모든 코드 작성, 빌드, 실행은 Docker 컨테이너 안에서 이루어지는 것을 전제로 한다.
- 패키지 설치는 반드시 Docker 컨테이너 내부에서 처리한다 (로컬 직접 설치 X).
- 코드를 줄 때는 항상 해당 파일이 프로젝트 구조 내에서 어디에 위치하는지 경로를 명시한다.
- Dockerfile은 multi-stage build로 작성한다. dev/prod 스테이지를 분리하고 production 이미지는 최소 사이즈를 목표로 한다.

## 기술 스택 (변경하지 말 것)
- 프레임워크: Next.js 15 (App Router, TypeScript)
- 스타일: Tailwind CSS 4
- 상태관리: Zustand
- 다국어: next-intl (한국어/영어)
- 컨테이너: Docker + Docker Compose
- 게임 데이터: 정적 JSON 파일 (DB 사용하지 않음)
- 계산 엔진: 클라이언트 사이드 TypeScript (서버 API 호출 X)
- 배포 대상: GCP Compute Engine (Docker)

## 코드 작성 규칙
- 언어: TypeScript strict mode
- 컴포넌트: React 함수형 컴포넌트 + hooks
- 스타일: Tailwind 유틸리티 클래스만 사용, 별도 CSS 파일 작성 금지
- 네이밍: 컴포넌트 PascalCase, 함수/변수 camelCase, 타입/인터페이스 PascalCase
- 파일명: 컴포넌트는 PascalCase.tsx, 로직/유틸은 camelCase.ts
- 대미지 계산 관련 함수는 순수 함수(pure function)로 작성, React에 의존하지 않게 한다
- 게임 데이터 JSON은 src/data/ 아래에 놓고, TypeScript 타입을 반드시 정의한다
- 매직 넘버 금지 — 대미지 공식의 상수는 반드시 이름 붙인 상수로 정의한다
- 모든 export 함수에는 JSDoc으로 파라미터와 리턴 타입의 의미를 기술한다 (타입 자체로 충분하면 생략 가능)
- any 타입 사용 금지. unknown + 타입 가드로 처리한다

## 파일 헤더 규칙
- 모든 개발 파일(.ts, .tsx, .css 등) 최상단에 주석으로 해당 파일의 경로를 적는다.
- 형식 예시:
  - TypeScript/React: `// src/lib/calc/damage.ts`
  - CSS: `/* src/app/globals.css */`

## 네이밍/주석 품질 규칙 (엄격히 준수)
- 모든 변수명, 함수명, 메서드명, 타입명, 파일명은 실무 시니어 개발자가 코드리뷰에서 통과시킬 수준으로 작성한다.
- 억지 네이밍 금지. 의미가 즉시 파악되는 명확한 이름만 사용한다.
  - 나쁜 예: `calcDmgV2`, `processDataFinal`, `helperFunc`, `doStuff`, `myHandler`, `xxxManager`
  - 좋은 예: `calcExpectedDamage`, `sumSubStats`, `formatPercent`, `parseEchoInput`
- 버전 접미사 절대 금지. `v2`, `V5`, `New`, `Final`, `_old`, `_backup` 같은 접미사를 변수명/함수명/파일명 어디에도 붙이지 않는다. 리팩토링이 필요하면 기존 것을 직접 수정한다.
- 주석도 동일 기준 적용:
  - 나쁜 예: `// 여기서 처리함`, `// TODO: 나중에 고치기`, `// ver3 로직`, `// 새로운 버전`
  - 좋은 예: `// 에코 서브스탯을 최종 ATK에 합산`, `// 적 DEF 기반 대미지 감소 계수`
- 주석은 "무엇을 하는지"가 아니라 "왜 이렇게 하는지"를 설명할 때만 작성한다. 코드 자체로 의도가 명확하면 주석을 달지 않는다.
- 불필요한 주석 금지: `// 변수 선언`, `// 리턴`, `// 함수 끝` 같은 자명한 주석은 절대 작성하지 않는다.

## 대화 규칙
- 코드를 줄 때 파일 경로를 항상 주석 또는 코드블록 위에 표시한다
- 긴 코드는 파일 단위로 나눠서 준다 (한 메시지에 파일 3개 이하 권장)
- 게임 메커니즘 관련 가정을 할 때는 반드시 근거를 명시한다
- 프로젝트 지식 파일에 있는 기획서/공식 문서를 참조하되, 게임 업데이트로 변경되었을 수 있는 수치는 확인을 요청한다
- 기능을 구현할 때 "간략화 버전"이나 "단순화 버전"을 임의로 만들지 않는다. 요청한 기능은 프로덕션 스펙 그대로 완전하게 구현한다.

## 현재 개발 단계
Phase 1 (MVP): 에코 비교 계산기 + 무기 비교

- 무기 교체 시 딜 변화 가시화
- 반응형 UI (모바일 우선), 다크모드 기본
- 한국어/영어 2개 언어
